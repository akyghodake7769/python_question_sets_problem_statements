================================================================================
               KODEARENA AWS_Q22_E LAB: STEP-BY-STEP TESTING GUIDE
================================================================================

This guide outlines the workflow required to successfully complete and test 
the AWS_Q22_E (AWS Application Load Balancer Path-Based Routing) lab in KodeArena 
on your AWS environment. Following these steps ensures 100% compliance with the 
real-time driver audit.

IMPORTANT: To avoid resource name collisions, always PREPEND your unique AWS IAM 
username (e.g., `ltm-devops-user1`) to the front of all provisioned EC2 instances, 
Application Load Balancers, Security Groups, and Target Groups using the conventions below.
Do NOT use "labskraft" or your platform email username as a suffix.

--------------------------------------------------------------------------------
PHASE 1: PROVISION EC2 INSTANCES & CONFIGURE WEB SERVERS (Satisfies TC1 & TC5)
--------------------------------------------------------------------------------
1. Launch Three EC2 Instances (Ubuntu 22.04 LTS):
   - Instance 1:
     * Name Tag: `<your-aws-iam-username>-service-a-host`
     * Availability Zone: Zone 1 (e.g., `us-east-1a`)
   - Instance 2:
     * Name Tag: `<your-aws-iam-username>-service-b-host`
     * Availability Zone: Zone 2 (e.g., `us-east-1b`)
   - Instance 3:
     * Name Tag: `<your-aws-iam-username>-service-c-host`
     * Availability Zone: Zone 3 (e.g., `us-east-1c`)
   
   Ensure that all three instances are running and spread across three distinct 
   Availability Zones (AZs) in the same region.

2. Install Nginx and Configure Content on Each Instance:
   - Connect to `<your-aws-iam-username>-service-a-host` via SSH:
     sudo apt update && sudo apt install nginx -y
     sudo mkdir -p /var/www/html/app1
     echo "Product Catalog Service - Status: OK" | sudo tee /var/www/html/app1/index.html
     sudo systemctl restart nginx
     
   - Connect to `<your-aws-iam-username>-service-b-host` via SSH:
     sudo apt update && sudo apt install nginx -y
     sudo mkdir -p /var/www/html/app2
     echo "Order Management Service - Status: OK" | sudo tee /var/www/html/app2/index.html
     sudo systemctl restart nginx
     
   - Connect to `<your-aws-iam-username>-service-c-host` via SSH:
     sudo apt update && sudo apt install nginx -y
     sudo mkdir -p /var/www/html/app3
     echo "User Profile Service - Status: OK" | sudo tee /var/www/html/app3/index.html
     sudo systemctl restart nginx

--------------------------------------------------------------------------------
PHASE 2: SECURE PORT 80 VIA SECURITY GROUPS (Satisfies TC4)
--------------------------------------------------------------------------------
1. Create the Application Load Balancer Security Group:
   - In the EC2 console, go to Network & Security -> Security Groups -> Create Security Group.
   - Security Group Name: `<your-aws-iam-username>-alb-sg`
   - Description: Security group for Application Load Balancer.
   - Inbound Rules:
     * Add Rule: Type: HTTP (Port 80), Source: Anywhere-IPv4 (0.0.0.0/0).
   - Click Create Security Group.

2. Create/Update the EC2 Instances Security Group:
   - Go to Network & Security -> Security Groups -> Create Security Group.
   - Security Group Name: `<your-aws-iam-username>-instances-sg`
   - Description: Security group for microservice instances.
   - Inbound Rules:
     * Add Rule: Type: HTTP (Port 80), Source: Custom -> Select the Security Group ID of `<your-aws-iam-username>-alb-sg`.
     * Add Rule: Type: SSH (Port 22), Source: My IP (for administrative SSH access).
   - Click Create Security Group.
   - Associate this `<your-aws-iam-username>-instances-sg` security group with all three EC2 instances. Remove any default security group that allows public port 80 traffic.

--------------------------------------------------------------------------------
PHASE 3: CREATE TARGET GROUPS (Satisfies TC3)
--------------------------------------------------------------------------------
1. Create Target Group for Service A:
   - Go to EC2 -> Target Groups -> Create target group.
   - Target type: Instances.
   - Target Group Name: `<your-aws-iam-username>-tg-app1`
   - Protocol: HTTP, Port: 80.
   - Health checks:
     * Protocol: HTTP.
     * Health check path: `/app1/` (or `/app1/index.html`).
   - Click Next. Select `<your-aws-iam-username>-service-a-host`, click "Include as pending below", and click "Create target group".

2. Create Target Group for Service B:
   - Go to EC2 -> Target Groups -> Create target group.
   - Target type: Instances.
   - Target Group Name: `<your-aws-iam-username>-tg-app2`
   - Protocol: HTTP, Port: 80.
   - Health checks:
     * Protocol: HTTP.
     * Health check path: `/app2/` (or `/app2/index.html`).
   - Click Next. Select `<your-aws-iam-username>-service-b-host`, click "Include as pending below", and click "Create target group".

3. Create Target Group for Service C:
   - Go to EC2 -> Target Groups -> Create target group.
   - Target type: Instances.
   - Target Group Name: `<your-aws-iam-username>-tg-app3`
   - Protocol: HTTP, Port: 80.
   - Health checks:
     * Protocol: HTTP.
     * Health check path: `/app3/` (or `/app3/index.html`).
   - Click Next. Select `<your-aws-iam-username>-service-c-host`, click "Include as pending below", and click "Create target group".

--------------------------------------------------------------------------------
PHASE 4: PROVISION ALB & CONFIGURE PATH LISTENER RULES (Satisfies TC2 & TC3)
--------------------------------------------------------------------------------
1. Create the Application Load Balancer:
   - Go to EC2 -> Load Balancers -> Create load balancer -> Application Load Balancer.
   - Name: `<your-aws-iam-username>-services-alb`
   - Scheme: Internet-facing
   - IP address type: IPv4
   - Network mapping: Select your VPC and check at least three Availability Zones (matching the AZs where the instances are deployed).
   - Security groups: Select `<your-aws-iam-username>-alb-sg`.
   - Listeners and routing:
     * Protocol: HTTP, Port: 80.
     * Default action: Forward to `<your-aws-iam-username>-tg-app1`.
   - Click Create load balancer. Wait for the state to transition to Active.

2. Configure Path-Based Routing Rules:
   - Select the `<your-aws-iam-username>-services-alb` load balancer.
   - Go to the Listeners tab, select the HTTP:80 listener, and click "View/edit rules" (or "Manage listener" -> "Edit rules").
   - Click "+" or "Add rules". Define rules with these path patterns:
     * Rule 1:
       - Condition: Path is `/app1*`
       - Action: Forward to `<your-aws-iam-username>-tg-app1`
     * Rule 2:
       - Condition: Path is `/app2*`
       - Action: Forward to `<your-aws-iam-username>-tg-app2`
     * Rule 3:
       - Condition: Path is `/app3*`
       - Action: Forward to `<your-aws-iam-username>-tg-app3`
   - Click Save to apply listener rules.

--------------------------------------------------------------------------------
PHASE 5: RUN REAL-TIME EVALUATION
--------------------------------------------------------------------------------
1. Run Tests in KodeArena:
   - Open your browser to the ALB DNS name to test manually:
     * http://<alb-dns-name>/app1/ should return "Product Catalog Service - Status: OK"
     * http://<alb-dns-name>/app2/ should return "Order Management Service - Status: OK"
     * http://<alb-dns-name>/app3/ should return "User Profile Service - Status: OK"
   - Go back to the KodeArena IDE and click "Run Tests" (or execute `python student_workspace/run.py`).
   - The test suite will inspect the EC2 hosts, Availability Zones, Application Load Balancer, Listener Rules, Target Group memberships, and Instance Security Groups to verify the security and routing flow. It will report a perfect 20/20 [PASSED] score!
